| 2026-06-06 | Localization | Resolve Livewire title attributes as translation keys through the shared head partial when a matching language line exists. | PHP attributes cannot call translation helpers directly, and the shared layout can translate keys without changing Livewire routes. |

| 2026-06-06 | Private workspace boundary | Keep the owning `User` as the private workspace boundary for Step 017 and do not add a separate `Workspace` model yet. | Current todo, project, and tag resources already have direct `user_id` ownership; a separate workspace table would add migration and authorization scope before collaboration/member features exist. |
| 2026-06-06 | Dashboard privacy | Read dashboard counters through `DailySummaryQuery::for($user)` and owner-scoped query objects. | Dashboard counts are private data and must follow the same scoped query boundary as todo lists, search, filters, and project/tag pickers. |
| 2026-06-06 | Tampered filters | Return an empty scoped todo result for foreign, archived, or missing project/tag filter IDs. | Ignoring a bad numeric filter could fall back to a broader list, while applying it could let another user's ID influence the current user's query. Empty results preserve privacy and make the tampered state obvious. |
| 2026-06-06 | Authorization roles | Do not add placeholder viewer/editor/manager/owner roles during Step 019. | The current app has no workspace membership model; adding role stubs before collaboration would create false policy coverage and likely conflict with the planned invite/member steps. |
| 2026-06-06 | Request validation tests | Keep translated Form Request contract tests in the feature suite. | The unit suite does not boot Laravel's translator container, and these request classes intentionally expose translated attributes and messages. |
| 2026-06-06 | Email verification enforcement | Implement `MustVerifyEmail` on `User` and keep dashboard, todos, appearance, setup, security, and maintenance behind `verified`; leave profile settings auth-only. | Routes and docs already required verified private pages, but Laravel's verified middleware only enforces that contract for models implementing `MustVerifyEmail`. Profile remains reachable so unverified users can see their verification status and manage account details. |
| 2026-06-06 | Core task creation boundary | Keep task creation in `CreateTodo`, assign ownership only through the user's relationship, trim titles at the action write boundary, re-scope organization ids inside the action, and keep lifecycle/system fields out of todo mass assignment. | Livewire validation improves feedback, but backend callers must still be safe. Creation should not be able to complete, archive, delete, or cross-link a task by mass assignment or forged ids. |
| 2026-06-06 | Private task detail lookup | Implement task detail pages as class-based Livewire pages that accept a numeric id but resolve the task through `TodoListQuery::findVisibleFor()` instead of implicit model binding. | A raw implicit binding would load the record before owner scoping. The query object gives the same not-found behavior used by list actions and avoids leaking another user's title or metadata. |
| 2026-06-06 | Task editing boundary | Keep task editing in `UpdateTodo`, trim titles at the action write boundary, re-scope project/tag ids inside the action, reject archived edits, and keep completion/archive/delete state in separate lifecycle actions. | Editing should update details only. Lifecycle transitions need their own authorization, events, tests, and user feedback so the app does not blur complete, archive, restore, and delete behavior. |
| 2026-06-06 | Task completion boundary | Use explicit `CompleteTodo` and `ReopenTodo` action classes, `TodoCompleted` and `TodoReopened` events, and state-specific Livewire methods while preserving the compact row checkbox UI. | A generic toggle action hid two different lifecycle transitions and one generic event was not enough for later activity/reminder rules. Separate transitions keep policy, events, labels, and tests precise without changing the fast UI. |
| 2026-06-06 | Task archive return path | Name task archive reversal as unarchive in actions, policy abilities, Livewire methods, translations, and bulk workflows. | The plan step title says archive and restore, but task archive reversal is not the same as future soft-delete restore. Explicit unarchive naming avoids mixing archive reversal with trash recovery while preserving project restore behavior. |
| 2026-06-06 | Task trash behavior | Support restore from Trash with `RestoreDeletedTodo`, `TodoRestoredFromTrash`, `TodoPolicy::restore`, and owner-scoped `findTrashedFor()` while keeping `forceDelete` denied. | Soft delete recovery is useful and safe for restricted hosting, but permanent deletion needs a stricter product and authorization design before exposing irreversible controls. |
| 2026-06-06 | Lifecycle state machine | Keep lifecycle transition rules in `TodoTransition` and `TodoLifecycleStateMachine`, with action classes delegating guard decisions before mutation. | Separate action classes remain readable, while one transition map prevents complete/archive/delete/restore meanings from drifting apart as later steps add more task surfaces. |
| 2026-06-06 | Project detail routing | Implement project detail as a class-based Livewire page that accepts a numeric id but resolves through `ProjectListQuery::findVisibleFor()`. | Implicit project binding would load a project before owner scoping. The query boundary gives the same not-found privacy behavior used by task detail pages and keeps archived projects reviewable without making them assignment targets. |
| 2026-06-06 | Tag browsing | Make tag badges link to the existing owner-scoped todo tag filter instead of adding a separate tag detail page or bulk tag workflow in Step 029. | The current requirement is scoped labels, assignment, filtering, and validation. A filter link keeps labels useful now while leaving bulk tag assignment and richer label views to their later dedicated workflow steps. |
| 2026-06-06 | Priority validation | Use Laravel's backed-enum validation rule for task priorities and keep direct DTO priority parsing strict when a value is provided. | The enum already owns the allowed priority values. Validation should reject unexpected values at the UI boundary and direct action boundary instead of letting invalid data silently become Normal. |
| 2026-06-06 | Priority sorting | Generate priority sort SQL from `App\Enums\Priority` weights instead of hardcoding a separate query-only weight map. | Filtering, badges, translations, factories, seeders, and sorting should all mean the same priority order as later task surfaces are added. |
| 2026-06-06 | Due-date format | Store and accept task due dates as canonical `Y-m-d` date-only values through `App\Rules\Todos\DueDate`. | Browser date inputs, the database date column, factories, seeders, filters, and direct action calls need one strict format instead of accepting natural-language dates through PHP parsing. |
| 2026-06-06 | Due-date timezone | Compare due-date buckets in the configured app timezone, currently UTC, until per-user timezone settings are implemented. | The app has no user timezone field yet. A documented app-timezone contract is safer than pretending account-specific date boundaries exist. |
| 2026-06-06 | Today route | Add `todos.today` as a dedicated class-based Livewire page instead of only relying on `todos.index?due=today`. | The main filtered list remains useful, but a focused Today route gives dashboard/navigation targets and tests a stable current-task workflow while reusing the same owner-scoped query boundary. |
| 2026-06-06 | Today actions | Resolve Today page mutations through `TodoListQuery::findTodayFor()` instead of the broader `findVisibleFor()`. | A forged Livewire call should not let the focused Today page complete an owned task that is not active and due today, even though the main task workspace can handle that task elsewhere. |
| 2026-06-06 | Overdue route | Add `todos.overdue` as a dedicated class-based Livewire page instead of only relying on `todos.index?due=overdue`. | Overdue work needs a stable, focused review route and dashboard target while still reusing the same owner-scoped date query boundary as the main task list. |
| 2026-06-06 | Overdue actions | Resolve Overdue page mutations through `TodoListQuery::findOverdueFor()` instead of the broader `findVisibleFor()`. | A forged Livewire call should not let the focused Overdue page complete an owned task that is not active and past due. |
| 2026-06-06 | Upcoming route | Add `todos.upcoming` as a dedicated class-based Livewire page instead of only relying on `todos.index?due=upcoming`. | Future-dated work needs a stable planning route and dashboard target while still reusing the same owner-scoped date query boundary as the main task list. |
| 2026-06-06 | Upcoming actions | Resolve Upcoming page mutations through `TodoListQuery::findUpcomingFor()` instead of the broader `findVisibleFor()`. | A forged Livewire call should not let the focused Upcoming page complete an owned task that is not active and due after today. |
| 2026-06-06 | Search engine | Keep Step 035 search as owner-scoped, bound SQL `LIKE` with escaped wildcards instead of adding a hosted search service or full-text index. | The current app needs private title search, SQLite/local compatibility, and restricted-hosting safety without paid services, workers, or external daemons. |
| 2026-06-06 | Search query parameters | Treat non-numeric project/tag URL values as invalid filters that produce an empty result, matching foreign or missing numeric filter ids. | Ignoring an unexpected query parameter could widen a searched list; empty results preserve the user's explicit filter state without leaking private data. |
