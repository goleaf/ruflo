| 2026-06-05 | Request validation | Treat auth Form Request classes as the canonical rule source for Fortify registration and password reset actions. | Fortify passes arrays to action contracts, so sharing request-class rules avoids inline action validation while preserving Fortify's normal flow. |
| 2026-06-05 | Livewire validation | Keep Livewire-only task/profile validation in Livewire form objects or components unless a separate HTTP route/controller exists. | Livewire form objects are the correct boundary for component state and avoid forcing HTTP Form Requests into non-HTTP interactions. |
| 2026-06-05 | Ownership validation | Reject forged project, tag, and selected todo ids at the Livewire validation boundary while keeping action-level owner re-scoping. | Users get clear validation feedback and direct action calls remain safe if validation is bypassed. |
| 2026-06-05 | Future validation rules | Do not create placeholder invite, recurrence, reminder, upload, import/export, settings, or role rules before those stable domains exist. | Rules need concrete callers and tests; placeholders create false coverage and may fight later feature designs. Step 015 recheck removed the empty reminder-time placeholder rule. |
| 2026-06-06 | Localization | Resolve Livewire title attributes as translation keys through the shared head partial when a matching language line exists. | PHP attributes cannot call translation helpers directly, and the shared layout can translate keys without changing Livewire routes. |
| 2026-06-06 | Progress tracking | Expand root progress and checklist files to one explicit line per step from 001 through 100. | The current prompt forbids compressed ranges and requires every step to have its own status trail. |
| 2026-06-06 | Setup diagnostics | Store only generic database inspection failure markers in public Livewire setup status state. | Livewire serializes public state into the browser snapshot, so raw connection exceptions must not be exposed even on protected setup pages. |
| 2026-06-06 | Maintenance authorization | Use `users.is_admin` and an `access-maintenance-center` gate for the protected maintenance center until the later workspace/member role system exists. | Step 008 requires normal users to be denied, and the current app has no workspace or role model yet. |
| 2026-06-06 | Private workspace boundary | Keep the owning `User` as the private workspace boundary for Step 017 and do not add a separate `Workspace` model yet. | Current todo, project, and tag resources already have direct `user_id` ownership; a separate workspace table would add migration and authorization scope before collaboration/member features exist. |
| 2026-06-06 | Dashboard privacy | Read dashboard counters through `DailySummaryQuery::for($user)` and owner-scoped query objects. | Dashboard counts are private data and must follow the same scoped query boundary as todo lists, search, filters, and project/tag pickers. |
| 2026-06-06 | Tampered filters | Return an empty scoped todo result for foreign, archived, or missing project/tag filter IDs. | Ignoring a bad numeric filter could fall back to a broader list, while applying it could let another user's ID influence the current user's query. Empty results preserve privacy and make the tampered state obvious. |
| 2026-06-06 | Authorization roles | Do not add placeholder viewer/editor/manager/owner roles during Step 019. | The current app has no workspace membership model; adding role stubs before collaboration would create false policy coverage and likely conflict with the planned invite/member steps. |
| 2026-06-06 | Request validation tests | Keep translated Form Request contract tests in the feature suite. | The unit suite does not boot Laravel's translator container, and these request classes intentionally expose translated attributes and messages. |
| 2026-06-06 | Email verification enforcement | Implement `MustVerifyEmail` on `User` and keep dashboard, todos, appearance, setup, security, and maintenance behind `verified`; leave profile settings auth-only. | Routes and docs already required verified private pages, but Laravel's verified middleware only enforces that contract for models implementing `MustVerifyEmail`. Profile remains reachable so unverified users can see their verification status and manage account details. |
| 2026-06-06 | Core task creation boundary | Keep task creation in `CreateTodo`, assign ownership only through the user's relationship, trim titles at the action write boundary, re-scope organization ids inside the action, and keep lifecycle/system fields out of todo mass assignment. | Livewire validation improves feedback, but backend callers must still be safe. Creation should not be able to complete, archive, delete, or cross-link a task by mass assignment or forged ids. |
| 2026-06-06 | Private task detail lookup | Implement task detail pages as class-based Livewire pages that accept a numeric id but resolve the task through `TodoListQuery::findVisibleFor()` instead of implicit model binding. | A raw implicit binding would load the record before owner scoping. The query object gives the same not-found behavior used by list actions and avoids leaking another user's title or metadata. |
| 2026-06-06 | Task editing boundary | Keep task editing in `UpdateTodo`, trim titles at the action write boundary, re-scope project/tag ids inside the action, reject archived edits, and keep completion/archive/delete state in separate lifecycle actions. | Editing should update details only. Lifecycle transitions need their own authorization, events, tests, and user feedback so the app does not blur complete, archive, restore, and delete behavior. |
| 2026-06-06 | Task completion boundary | Use explicit `CompleteTodo` and `ReopenTodo` action classes, `TodoCompleted` and `TodoReopened` events, and state-specific Livewire methods while preserving the compact row checkbox UI. | A generic toggle action hid two different lifecycle transitions and one generic event was not enough for later activity/reminder rules. Separate transitions keep policy, events, labels, and tests precise without changing the fast UI. |
